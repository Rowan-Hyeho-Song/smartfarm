베리디시전 · 모델링 작업 이력

모델링 작업 이력 — 원인 · 변경 · 이유

딸기 생육 예측 모델을 만들며 내린 핵심 모델링 결정과 실험을 "무엇이 문제였고, 어떻게 바꿨으며, 왜 그렇게 했는지"로 정리했습니다. (환경 셋팅·전처리 같은 준비 단계는 제외하고 모델링 중심으로만.)

범위 검증 · 앙상블 · 불확실성 · 튜닝 · 실험 핵심 결과 착과수 v2 R² 0.87 · 구간 적중률 83% 데이터 온실 1~4 · 2024-11~2025-02
전체 흐름 ① 검증방식 ② 앙상블·CV ③ 불확실성 ④ 튜닝 ⑤ 추론·구간 ⑥ 지표·창 실험 ⑦ 강규제 비교 ⑧ 룰 베이스 제어 ⑨ A/B 제어 실험 ⑩ 환경 예보 ⑪ 실증 계획 ⑫ v2 정확도 돌파 결과 · 파일

한눈에 — 모델링 핵심 12가지

  1. 검증방식 매칭 — 착과수는 '온실별 검증' + 발달단계 feature로 R² −1.3 → 0.67 (검증이 안 맞으면 "예측 불가"라는 오판).
  2. 앙상블 + 교차검증 — LightGBM+CatBoost 평균, 온실별 4-fold로 안정화 (0.62 → 0.66).
  3. 불확실성 정량화 + Conformal — 예측 구간 적중률 54% → 77%(목표 80%).
  4. Optuna 튜닝 — 설정값 자동 탐색으로 0.62 → 0.67 (온실4 홀드아웃 0.73).
  5. 추론 + 예측 구간 — 저장 모델로 제출, split-conformal로 구간 폭 5.0 → 3.63(적중률 83%).
  6. 지표·창 실험 — 7개 생육 지표 × 검증방식 × 집계 창을 비교해 지표별 최적 규명.
  7. 강규제 선형 vs 부스팅 — Ridge/Lasso를 실측 비교, 우리 데이터엔 부스팅이 우세.
  8. 룰 베이스 제어 설계 — 데이터 상관(복합·다변량)으로 제어 규칙 지정. CO₂가 유일한 독립 레버, VPD·습도는 그림자.
  9. A/B 제어 실험(CO₂·온도) — 고CO₂ 초장 +1.57cm(p<0.001·포화형). 온도는 저온일수록 신장↑이나 '도장' 트레이드오프 레버.
  10. 환경 예보 트랙 — 내부온도 예보 성공(6h R²0.86), CO₂는 예보 불가(=조작 레버). 선제 제어의 눈.
  11. 실증 계획서 — CO₂ 무작위 배정 A/B 프로토콜 4단계 작성(온실 확보 시 착수).
  12. v2 정확도 돌파 — 예측 단위 '개체→온실-날짜 평균' + lag(전주 생육) + 튜닝 → 착과수 R² 0.87(80% 돌파).

1

검증방식이 안 맞으면 예측이 안 된다 — 착과수

모델
원인
첫 베이스라인에서 착과수 R²=−1.33(평균 찍기보다 못함). ① 착과수는 '직전 환경'보다 화방 발달주기(재배 몇 주차)에 좌우됨. ② 데이터가 15주뿐이라 '뒤 4주 검증(시간분할)'은 그 주차가 학습범위 밖 → 트리 외삽 불가.
변경
A. 발달단계 feature(재배경과주차·조사회차·월) 추가. B. 검증방식 스위치(VALIDATION) — time(뒤 N주) / greenhouse(온실 하나를 통째로 검증). 착과수는 greenhouse. (평가 시 미래 정보가 새지 않게 조사 당일 환경은 제외 = 누수 차단)
이유
'온실별 검증'은 모든 주차가 학습에 포함되어 발달단계 feature가 힘을 씁니다. → R² 0.67.
착과수 · 검증방식featureR²
시간분할(뒤4주)환경만−1.20
시간분할(뒤4주)환경+시점−1.41
온실별(LOGO)환경만+0.59
온실별(LOGO)환경+시점+0.62
온실별(온실4 검증)환경+시점 ← 최종 채택+0.67
같은 데이터·같은 모델이어도 검증방식을 타깃 성격에 맞추지 않으면 "예측 불가"라는 잘못된 결론에 빠집니다. 착과수(발달주기 지배)는 온실별, 초장·엽폭(환경 따라 서서히)은 시간분할이 맞습니다.
2

단일 모델·단일 검증의 우연을 줄이다

모델
원인
모델 하나·검증 한 번은 운에 흔들립니다. 점수가 진짜 실력인지 우연인지 구분이 안 됨.
변경
LightGBM + CatBoost 앙상블(두 모델 예측 평균)과 온실별 4-fold 교차검증(온실 하나씩 빼며 4번) 적용. (train_advanced.py)
이유
서로 다른 모델의 실수가 상쇄돼 더 안정적. 착과수 R² 0.62 → 0.65(CatBoost) → 0.66(앙상블).
3

값 하나가 아니라 '얼마나 확신하나'까지

모델
원인
의사결정엔 예측값뿐 아니라 불확실성이 필요. 분위수 회귀로 만든 예측 구간(P10~P90)의 적중률이 54%로 목표 80%에 한참 못 미침(과신).
변경
Conformal 보정(CQR) — 보정용 데이터에서 '구간이 빗나간 양'을 재서 그만큼 구간을 넓힘. (conformal.py)
이유
구간이 약속한 80%만큼 실제로 맞도록 교정하기 위해. 적중률 54% → 77%(구간이 넓어진 건 '정직해진 것').
4

설정값을 손이 아니라 Optuna로

모델
원인
트리 깊이·학습률 등 하이퍼파라미터의 기본값이 최적이라는 보장이 없음.
변경
Optuna로 60회 자동 탐색(LOGO R²를 기준으로). 최적 설정으로 재학습·저장. (train_optuna.py)
이유
수동보다 빠르고 체계적. 착과수 LOGO R² 0.62 → 0.67, 온실4 홀드아웃 0.73 (튜닝 단일모델이 앙상블도 상회).
5

학습만 있고 '사용(제출)'이 없었다

모델
원인
파이프라인에 추론·제출 단계가 없음. 처음 만든 구간은 방식 오류로 과하게 넓었음(폭 5.0, 적중률 96%).
변경
predict.py — 튜닝 모델 로드 → 예측값+구간 출력. 정석 split-conformal(Q를 잰 모델과 예측 모델을 동일화) + 착과수 하한 0 클리핑.
이유
실전 제출 완성 + 더 좁은 구간. 폭 5.0 → 3.63, 적중률 96% → 83%(목표 80% 근접).
6

"왜 착과수·초장만?", "왜 7일?" — 데이터로 답하다

실험
원인
대표 지표 2개만 본 근거, 집계 창 7일의 근거를 설명할 수 있어야 함(발표 대비).
변경
7개 생육 지표 × 2검증, 집계 창 3/7/14일 전수 비교. (compare_experiments.py)
이유
지표가 두 부류로 갈림을 확인 — 온실별(착과수·관부직경·엽수) vs 시간분할(엽병장·엽폭·초장·엽장). 창은 착과수=14일, 크기지표=7일이 최적.
7

"데이터 적으면 강규제 선형" — 정말 그런가

실험
원인
데이터가 적을 때 강규제 선형(Ridge/Lasso)이 낫다는 통념을 우리 데이터로 검증할 필요.
변경
Ridge·Lasso vs LightGBM을 같은 조건으로 실측 비교.
이유
우리 환경→생육 관계는 비선형이라 선형이 약함. 착과수 Lasso 0.42 vs 부스팅 0.62 → 부스팅 채택. (Lasso는 feature 선택용으로만 유용)
8

예측을 넘어 '제어'로 — 상관 기반 룰 설계

제어
원인
폐루프 제어를 하려면 "이 조건이면 이렇게"라는 규칙이 필요. 그런데 원시 상관은 계절 교란으로 가짜(예: "온도↓ → 착과↑"는 겨울작 착시). 게다가 짝(pairwise) 상관은 변수 간 복합 효과를 못 가름.
변경
A. 발달단계(조사회차) 제거 후 상관으로 계절 교란 차단. B. 다변량 분석 — 부분상관(다른 변수 통제) + 상호작용으로 진짜 레버 규명. 이를 조건부 룰 세트로 정리.
이유
CO₂만 독립 레버(부분상관 +0.26)로 살아남고, VPD·습도는 CO₂·온도의 그림자(≈0)임을 확인 → 짝 상관만 믿는 오판 방지. 착과수는 환경 상관이 미미해 환경 룰을 지양(발달·적과로 관리).
예측 모델(berry_ai)이 "이대로면 생장이 이 정도"를 알려주면, 이 룰이 "그럼 CO₂·온도를 이렇게"라는 행동을 정합니다. 상세는 docs/control/룰베이스_제어규칙.html.
9

룰의 인과를 A/B로 검증 — CO₂ 시비

제어
원인
룰(상관)은 가설일 뿐. CO₂가 정말 생장을 늘리는지 인과 검증이 필요한데, 살아있는 온실이 없어 실제 무작위 실험은 불가.
변경
데이터로 A/B를 대리 재현 — A. 관측 준-A/B(같은 단계에서 고CO₂ vs 저CO₂, t검정) B. 모델 반사실 시뮬(CO₂ ±Δ 조작 시 예측 생장 변화, dose-response).
이유
두 방법 모두 CO₂↑ → 생장↑ 확인(초장 +1.57cm, p<0.001). 반사실은 비대칭 포화 반응(떨어뜨리면 −1.4cm, 올리면 이득 체감) → "적정선 유지·하락 방지"가 핵심 메시지.
상관(가설) → 대리 실험(검증)으로 한 단계 나아갔습니다. 진짜 인과 확정은 온실 무작위 배정 실증이 필요(문서에 프로토콜 포함). 상세는 docs/experiments/AB제어실험_CO2.html.
10

제어의 눈 — 앞으로 h시간 뒤 환경 예보

제어
원인
폐루프 제어가 '지금'만 보면 늦음(한파는 오기 전에 대응해야). "앞으로 환경이 어떻게 변할지"를 아는 예보가 필요.
변경
시간별 데이터로 h시간 뒤 내부온도·CO₂ 예보 (부스팅 + 시차 feature, 지속성 기준선과 비교). (forecast_env.py)
이유
온도 예보 성공(6h R²0.86, 12h 0.61, 지속성 우세) → 룰의 IF 조건에 넣어 선제 제어. CO₂는 예보 실패(지속성 못 이김) = 날씨가 아니라 '제어 행동'에 좌우 → 조작 레버라는 방증.
11

가설 → 인과 확정을 위한 실증 계획

제어
원인
대리 A/B는 강한 가설일 뿐. 인과 확정은 무작위 배정 실험이 필요한데 현재 살아있는 온실이 없음.
변경
온실 확보 시 즉시 실행 가능한 실증 계획서 작성 — 가설·무작위 배정·처치 정의·측정·분석·리스크. (docs/plan/실증실험_계획서.html)
이유
실증 데이터가 현 파이프라인(data_prep→train)에 동일 스키마로 바로 연결되게 설계 → 실증이 곧 모델 업데이트로 이어짐.
이로써 예측 → 불확실성 → 튜닝 → 제어 설계 → 제어 검증(A/B) → 예보 → 실증 계획까지 하나의 서사가 완성됩니다.
12

정확도 80% 돌파 — v2 (평균 + lag + 튜닝)

모델
원인
v1은 개체(표본) 하나하나를 예측 → 식물마다 랜덤 편차가 커서 R²가 0.67~0.73에서 막힘. 80% 목표에 부족.
변경
A. 예측 단위를 '온실-날짜 평균'으로 재정의(개체 노이즈 제거). B. lag feature(전주·전전주 생육) 추가(data_prep.py). C. Optuna 재튜닝(train_agg.py).
이유
우리가 제어하는 건 개체가 아니라 온실 전체 → 평균 예측이 더 올바른 정의이자 더 정확. 착과수 LOGO R² 0.57 → 0.78 → 0.87(튜닝). 초장도 0.19 → 0.52. 최고 기여 feature = 착과수_전주.
착과수 예측 방식LOGO R²
v1 · 개체 예측 (환경+시점)0.57
v1 · 개체 + lag0.64
v2 · 온실-날짜 평균 (환경+시점)0.78
v2 · 평균 + lag0.84
v2 · 평균 + lag + Optuna ← 최종0.87
"개체 하나를 정확히"가 아니라 "온실 전체를 정확히"로 문제를 올바로 정의하니 정확도가 뚫렸습니다. 단, 집계라 60행(온실×주)뿐이라 데이터가 늘면 재확인 권장. (다음: predict.py를 v2 경로로 연결)
