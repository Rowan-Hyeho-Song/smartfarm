딸기 의사결정 AI를 위한
오픈 모델 지형 & 혼합운용 리서치

보유 중인 4개 온실의 환경·생육 데이터로 "목표 기반 농작업 의사결정형 딸기 AI"를 만들 때, 지금(2025–2026) 쓸 수 있는 오픈 모델은 무엇이 있고, 각각의 장단점과 라이선스는 어떠하며, 이들을 어떻게 혼합운용(앙상블·하이브리드) 하는지 정리한 문서입니다.

한눈 요약 (TL;DR)

  • "하나의 만능 모델"은 없습니다. 역할별로 다른 오픈 모델을 층층이 쌓아 혼합운용하는 것이 정답입니다 — 회의록의 방향과 일치합니다.
  • 환경 센서 예측: 오픈·상용가능·다변량(공변량) 지원의 Amazon Chronos-2가 1순위. 단, 소규모·단기 예측에선 LightGBM + 기본 베이스라인이 더 나을 때가 많아 반드시 비교.
  • 생육·수확 예측(정형 데이터): CatBoost / LightGBM(부스팅 트리)이 여전히 강자. 소량 데이터엔 TabPFN v2를 함께 벤치마크. 시계열 딥러닝(TFT 등)은 데이터가 많아질 때.
  • 자연어·에이전트·교차검증(LLM): 한국어+상용가능+에이전트 관점에서 Qwen3 + 카카오 Kanana-2 조합이 안전한 백본. 무거운 추론엔 DeepSeek-R1(MIT). LG EXAONE는 성능은 최상급이나 연구용 라이선스라 상용은 별도 계약 필요.
  • 혼합운용의 핵심: 물리모델(딸기 sink–source) + ML 하이브리드, 다중모달 late fusion, 파운데이션+로컬 잔차보정, LLM 오케스트레이션 + GraphRAG(온톨로지 접지) + 가드레일, 그리고 사람 검토(휴먼 인 더 루프).
  • 1차 버전은 자동제어가 아니라 "점검·추천"이어야 합니다. RL Actor는 실행이 아니라 제안만 — 소량 데이터·안전 현실에 맞는 선택입니다.

출발점

회의록 아키텍처를 오픈 모델로 매핑하면

보완의견(회의록)이 제시한 엔진 구성요소 각각에, "지금 쓸 수 있는 실제 오픈 모델"을 대응시켰습니다. 표의 마지막 열이 이 리서치의 핵심 결론입니다.

카테고리 ①

시계열 파운데이션 모델 — 환경·센서 예측

수십억 개의 시계열로 미리 학습돼, 처음 보는 센서 데이터도 제로샷(무학습)으로 예측하는 대형 모델군입니다. 온·습도·CO₂·일사·VPD·EC/pH 같은 온실 센서 예측이 대상입니다.

Amazon Chronos-2 Apache-2.0 · 상용OK

오픈 시계열 파운데이션 중 제로샷 다변량 + 공변량(외생변수)을 기본 지원하는 거의 유일한 선택. "일사·관수·설정값을 주면 내부 온도·습도·VPD를 예측"에 정확히 맞습니다. 경량 Chronos-Bolt는 단일 센서 초단기 예측에 빠르고 저렴.

환경 예측의 1순위 후보. 제로샷으로 시작 → 4개 온실 데이터로 파인튜닝. 공변량 지원 덕에 "제어 입력 → 환경 반응" 구조에 직결됩니다.

Datadog Toto Apache-2.0

관측(텔레메트리) 지표 특화. 고빈도·다변량·노이즈·스파이크가 온실 IoT 센서와 매우 흡사 — 저평가된 강력 후보. 직접 벤치마크 권장.

TabPFN-TS Apache pkg · 가중치 확인

소량 데이터·짧은 이력에 강함(외생변수 지원). 온실별 데이터가 적을 때 유리. TabPFN 가중치 상용 라이선스는 확인 필요.

Google TimesFM 2.5 Apache-2.0

단변량 장기맥락 강자(맥락 16K). 일주기·계절 패턴 포착에 좋음. 공변량은 Chronos-2보다 약함.

Sundial / Time-MoE / Lag-Llama Apache-2.0

경량·확률예측 베이스라인. Sundial은 밀리초 확률예측(제어 불확실성 밴드에 유용), Lag-Llama는 CPU에서도 구동.

라이선스 함정. 성능이 좋은 Salesforce Moirai 1.x / Moirai-MoE는 CC-BY-NC(비상업)라 상용 제품엔 부적합(Moirai 2.0은 미확인 → 확인 필요). Nixtla TimeGPT는 성능·편의성은 좋으나 가중치 비공개 유료 API — 데이터가 외부로 나가고 온팜/엣지 배포에 제약.
중요한 현실: 분(minute) 단위·초단기 제어 예측에서는 파운데이션 모델이 단순 베이스라인(계절-나이브, LightGBM 지연특징)에 지는 경우가 흔합니다. 파운데이션 모델을 "무조건 우위"로 가정하지 말고, 항상 값싼 베이스라인과 비교(bake-off)한 뒤 채택하세요. 맥락 길이(context length)는 사실상 하이퍼파라미터라 튜닝이 필요합니다.

카테고리 ②

정형/시계열 ML — 생육·수확 예측

우리 데이터(주 단위 생육 측정 + 시간 단위 환경)는 전형적인 소~중 규모 정형 데이터입니다. 이 영역은 아직도 부스팅 트리가 왕입니다.

부스팅 트리: CatBoost · LightGBM · XGBoost MIT / Apache

정형 데이터의 사실상 표준. 수백~수천 행에서도 강하고, GPU 없이 빠르며, SHAP로 해석 가능. CatBoost는 범주형(품종·구역·개체ID) 처리와 기본값이 뛰어나 소량 데이터에서 튜닝 부담이 적음 → 종종 최고의 시작점.

초장·엽수·착과수·수확량 회귀의 1순위. 지연특징(lag)·누적광량(DLI)·생육도일(GDD)·정식 후 일수·품종·개체ID를 피처로. 개체·시즌 기준 그룹 교차검증으로 누수 방지.

TabPFN v2 Apache+귀속 · 상용확인

학습 없이 문맥으로 예측하는 정형 파운데이션(회귀 지원). ≤1만 행/500피처(2.5는 ~5만 행)의 소량 데이터에서 튜닝 없이 부스팅과 대등/우위. 우리 규모에 딱 맞아 필수 벤치마크 상대.

시계열 딥러닝: TFT·PatchTST·iTransformer·N-HiTS MIT/Apache

다수 시계열·장기 예측·많은 데이터가 있을 때 진가. TFT는 공변량·미래입력·구간예측·해석성이 강점이나 데이터 굶주림. 주 단위 생육 예측엔 과함 → 데이터 축적 후 고려.

"딥러닝이 항상 낫다"는 오해. 정형 데이터(≤1만 행)에서는 트리 앙상블이 딥러닝을 이긴다는 연구가 일관됩니다(Grinsztajn 2022, Shwartz-Ziv 2022). 작물 수확량 응용 문헌도 RF/XGBoost가 주류이고, 딥러닝은 영상이나 대규모 데이터에서 주로 유리합니다. DLinear 같은 초단순 선형 베이스라인이 트랜스포머를 이기기도 하니, 단순부터 시작하세요.
시작 레시피: (a) 생육/수확 회귀 → CatBoost(또는 LightGBM) + TabPFN v2 헤드투헤드, 선형/Ridge 베이스라인 포함. (b) 환경 다단계 예측이 필요하면 DLinear 베이스라인 후 필요 시 시계열 파운데이션·DL로 승급. 실제 데이터로 작은 bake-off가 유일하게 믿을 답입니다.

카테고리 ③

오픈 LLM — 자연어·에이전트·교차검증

LLM은 수치를 계산하는 게 아니라 ①다른 모델 출력 교차검증 ②한국어 설명·작업추천 ③에이전트 오케스트레이션(도구·ML 호출) ④온톨로지 접지(RAG) 역할입니다. 한국어 역량과 상용 라이선스가 결정적입니다.

Alibaba Qwen3 Apache-2.0

이 용도의 종합 1순위. 강한 한국어(119개 언어), 최고 수준 오픈 도구호출/에이전트(Qwen-Agent), 하이브리드 추론, 완전 상용 허용. 조밀 0.6~32B + MoE(30B-A3B/235B-A22B).

카카오 Kanana-2 Apache-2.0

한국어 네이티브 + 에이전트 특화 + 상용가능의 삼박자. 멀티턴 도구호출 3배 개선. 한국어 표현 품질이 필요한 설명·작업추천에 최적. (Dec 2025 공개)

DeepSeek-R1 / V3 MIT

최상급 추론(수학·논리)·완전 무료(MIT). 671B/37B-active MoE라 자가호스팅은 데이터센터급. 무거운 교차검증·상충신호 판정에 선택적 호출. Distill(7~70B)로 경량화 가능.

LG EXAONE 4.0 연구용 · 상용 별도계약

한국어 도메인 품질 최상급(KMMLU 등)이나 상용 불가(LG 협의 필요). 라이선스 확보 시에만 후보. 32B + 1.2B 온디바이스.

그 외 참고

Mistral Small 3.x / Magistral(Apache, 효율·유럽어 강·한국어 약) · Google Gemma 3(한국어 양호·다국어, 단 커스텀 Gemma 약관) · Microsoft Phi-4(MIT, 소형 추론 강자·한국어 약) · Meta Llama 3.3/4(커스텀 커뮤니티 라이선스, 한국어 비공식) · Ai2 OLMo(완전공개·감사용이·한국어 약).

권고 조합(한국어·상용·에이전트): 오케스트레이터/도구라우터 = Qwen3 14B~32B, 한국어 설명·작업추천 = Kanana-2(또는 Qwen3-32B), 무거운 교차검증 = DeepSeek-R1(공유 GPU 노드에서 선택 호출), 온톨로지 RAG = 14~32B 롱컨텍스트 모델. Qwen3 + Kanana-2(둘 다 Apache-2.0)가 가장 안전한 상용 한국어 백본입니다.
시계열-LLM 하이브리드: 시계열 파운데이션이 예측(점추정+분위수)을 JSON으로 요약 → LLM이 도구호출로 받아 한국어 추천+신뢰도 생성, 온톨로지로 접지. LLM에 원시 장문 시계열을 토큰으로 직접 먹이지 말 것 — 시계열 모델이 요약하게 하세요. (TS-Reasoner 등 2025 연구가 이 패턴을 뒷받침.)

카테고리 ④ (핵심 질문)

혼합운용 — 어떻게 섞어 쓰는가

"오픈 모델을 어떻게 혼합운용하는지"가 이 리서치의 핵심 질문이었습니다. 6가지 검증된 패턴을 우리 프로젝트 적용과 함께 정리합니다.

1. 고전 앙상블 — 부스팅 · 스태킹 · 가중평균

여러 모델을 합쳐 단일 모델을 능가. 정형 농업데이터에선 보통 부스팅이 우세. 스태킹은 베이스 모델들이 서로 다른 실수를 하고 데이터가 충분할 때만 이득 — 소량이면 메타레이어 과적합 위험.

Downstream 예측기(수확·병해·품질)에 적용. 먼저 역-RMSE 가중평균부터, 스태킹은 데이터가 쌓인 뒤. (한 연구의 R²=0.98은 단일연구·과대추정 가능 → 수치 약속 금물.)

2. 물리 + ML 하이브리드 (가장 가성비 높은 소량데이터 해법)

기계론적 작물모델(광합성·source–sink)과 ML을 결합 — ML이 잔차 보정하거나, 물리식이 신경망을 제약(PINN). 딸기 전용 sink–source 건물중 모델이 실제로 R²=0.97–0.99로 검증됨.

딸기 sink–source 모델을 "물리 사전(prior)"으로 수확 예측기에 투입하고, RL Actor의 월드모델로 활용. 물리 경계를 안전 가드레일로도 사용(물리적으로 불가능한 예측·행동은 자동 플래그). 소량 데이터 한계를 우회하는 최선의 경로.

3. 온실 제어를 위한 강화학습(RL) — 현실 점검

WUR 자율온실챌린지(세계 표준 벤치마크)의 우승팀들은 엔드투엔드 RL을 쓰지 않았습니다 — 작물균형 제어·비전+최적화가 사람 재배자를 이겼죠. 순수 RL 온실제어는 대부분 시뮬레이션 단계이고, 핵심 난제는 안전(되돌릴 수 없는 설정값으로 작물 손상 가능).

RL Actor는 v1에서 "실행"이 아니라 "제안"만. 물리 월드모델에 대해 안전 범위로 행동공간을 제약하고, 모든 제안을 검증 레이어→사람으로 라우팅. 오프라인/모델기반 RL 선호, 안전조건 감지기+규칙기반 폴백. 완전 자동제어는 안전 입증을 전제로 한 3단계 목표.

4. LLM + 전문모델 오케스트레이션

LLM이 수치를 계산하지 않고 전문 모델/도구를 지휘. 라우팅·오케스트레이터-워커 패턴, 도구는 MCP로 노출. MoE ≠ 멀티에이전트(MoE는 모델 내부 구조, 우리에게 필요한 건 멀티에이전트 오케스트레이션). LLM-as-judge는 사람과 80%+ 일치하나 순서·장황함·자기선호 편향이 있어 순서 교환(swap) 평가로 완화. 온톨로지 접지는 GraphRAG, 안전은 가드레일 + 휴먼 인 더 루프.

LLM 오케스트레이터가 (a) 예측기·RL Actor를 도구로 호출, (b) 온톨로지 GraphRAG로 근거 접지, (c) RL 제안을 LLM-judge로 교차검증(편향 보정), (d) 가드레일 통과 후 우선순위 작업목록을 사람에게. LLM은 계산이 아니라 지휘·접지·검증·설명.

5. 다중모달 융합 (센서+영상+생육+텍스트)

early/joint/late 3가지 중 v1은 late fusion: 각 모달 모델(CNN 병해, 시계열 예측, 생육 정형)을 따로 돌려 결정 레이어에서 결합. 결측 모달에 강인하고 각 모델이 감사 가능. 데이터가 늘면 intermediate로.

6. 시계열 파운데이션 + 로컬 잔차보정

제로샷 파운데이션을 단독 신뢰 금지. FM이 베이스라인 예측 → 로컬 LightGBM이 잔차 보정, 또는 역-RMSE 가중 블렌드. 소량 도메인에선 파인튜닝된 로컬 모델이 제로샷 FM을 이기는 사례가 일반적.

종합

제안: 5계층 참조 아키텍처

인지 → 예측 → 결정 → 검증 → 실행. 각 층에 어떤 모델이 앉고 어떻게 넘겨주는지. v1에서는 실행 직전에 사람 승인이 필수 게이트입니다.

1

인지 (Perception) 원시 → 구조화

센서 스트림·잎 영상·작업로그를 통합 피처테이블 + 타입 이벤트로 변환(late fusion 준비).

CNN 영상진단시계열 인제스트event2vec(=Berry2Vec)생육 정형피처
2

예측 (Prediction) 불확실성 포함

수확·병해·품질 예측 + 환경 시계열 예측. 물리모델이 예측·타당성 경계를 동시에 제공.

CatBoost/LightGBMTabPFN v2Chronos-2 + 로컬 잔차보정딸기 sink–source 물리모델
3

결정 (Decision) 제안만, 실행 X

RL Actor(오프라인·제약)가 설정값/작업 후보를 물리 월드모델로 평가·생성.

Offline/Model-based RL물리 월드모델안전 행동공간 제약
4

검증 (Validation) 접지·판정·가드레일

LLM 오케스트레이터가 LLM-judge 교차검증(편향보정) + 온톨로지 GraphRAG 접지 + 가드레일 + 물리경계 체크 → 우선순위 작업목록과 근거 생성.

Qwen3 / Kanana-2DeepSeek-R1(선택)GraphRAG가드레일
5

실행 (Action) v1=사람 승인

v1: 재배자 승인("점검·추천"). v3: 저위험·가역 설정값만 게이트 자동제어(안전감지기+규칙폴백). 결과는 다시 인지로 폐루프.

휴먼 인 더 루프결과 로깅 → event2vec
계층 간 인계는 불확실성을 담은 타입드 JSON으로. 어느 층이든 "신뢰도 낮음/모달 결측"을 플래그하고 우아하게 성능저하(graceful degrade) 하도록 설계.

실행

단계별 로드맵 (MVP → 고도화)

Phase 0
기반 — 데이터 + 물리사전

센서·영상·로그 정비, 딸기 온톨로지/지식그래프·이벤트 스키마 구축, sink–source 물리모델 세팅. 산출: 통합 피처스토어.

Phase 1 · MVP
자문형 (RL 없음, late fusion)

LightGBM 예측기 + FM 잔차 블렌드 + CNN 진단 + LLM 오케스트레이터(GraphRAG·가드레일)로 설명된 우선순위 추천. 100% 사람 검토, 자동제어 없음. 소량 데이터·안전에 맞는 안전한 v1.

Phase 2
제안엔진 추가 (섀도 모드)

RL Actor를 섀도(제안만, 미실행)로, LLM-judge 교차검증(swap) + 재배자 피드백 수집, 로그로 오프라인 RL 학습 시작.

Phase 3
게이트형 준자율

저위험·가역 설정값만 자동제어(안전감지기+규칙폴백+물리경계). 관수·급격한 기후변경 등 고위험·비가역은 사람 승인 유지. sim-to-real 검증하며 확대.

결론

핵심 권고 & 주의사항

이 리서치를 실행 지침 세 갈래로 요약하면:

  • 모델 선정 — 환경예측 Chronos-2(+베이스라인 비교), 생육/수확 CatBoost·LightGBM + TabPFN v2, 언어·에이전트 Qwen3 + Kanana-2(+선택적 DeepSeek-R1). 전부 상용 가능(Apache/MIT)으로 백본 구성.
  • 혼합운용 — 물리(딸기 sink–source)+ML 하이브리드, late fusion, FM+로컬 잔차보정, LLM 오케스트레이션+GraphRAG 접지, 사람 검토 게이트를 5계층으로 결합.
  • 안전·현실 — v1은 자동제어가 아닌 "점검·추천", RL은 제안만. 소량 데이터에선 파인튜닝된 로컬 모델·물리사전·사람 감독이 대형 제로샷/엔드투엔드 RL보다 낫다는 것이 일관된 결론.
표현·검증 주의(대외 문서화 시):
  • Berry2Vec는 공개 선행연구가 아닌 내부 명칭 — "event2vec 개념"으로 기술.
  • OpenTSLM은 의료·추론용이며 농업 검증·수치예측 이력 없음 — "설명 레이어(파인튜닝 필요)"로 위치.
  • 인용된 초고성능 수치(예: 물리+ML R²≈0.9995, 스태킹 R²=0.98)는 단일연구·과대추정 가능 — 목표수치로 약속 금물.
  • RL 에너지 40~58%↓·수확 +10%대 등은 대부분 시뮬레이션 결과.
  • 라이선스는 유동적(예: Moirai 2.0·Gemma 4·EXAONE 최신) — 배포 전 공식 모델카드 재확인.
