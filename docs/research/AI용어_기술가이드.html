01

큰 그림 — 지형과 우리 스택

단어들이 어디에 속하는지 지도부터.

흔히 뭉뚱그리는 세 단어의 포함 관계: AI ⊃ 머신러닝(ML) ⊃ 딥러닝(DL).

  • 머신러닝 = 데이터로 규칙을 스스로 학습. 우리가 쓰는 대부분.
  • 딥러닝 = 신경망(뉴럴넷) 기반 ML의 한 갈래. LSTM·트랜스포머가 여기.
  • 전통 ML(트리 계열) = 신경망 아님. 엑셀 같은 표(정형) 데이터에 강함. 우리 스택의 주력.
우리 팀이 실제로 쓰는 것

모델: RandomForest, LightGBM, XGBoost, CatBoost (전부 트리 계열). 검증: GroupKFold 교차검증. 지표: R²·MAE. 불확실성: CQR(등각예측) 신뢰구간. 구조: 환경 입력(X) → 생육 7개 예측(Y), 예측 + 농학 규칙 하이브리드. 회의에선 여기에 LSTM·트랜스포머·berry2vec를 얹을지 논의 중.

한 문장으로 ML의 전부

model.fit(X, y) = 문제(X)와 정답(y)으로 공부. model.predict(X) = 새 문제로 시험. 나머지는 전부 이 두 줄을 잘하기 위한 곁가지다.

02

모델 동물원

회의·코드에 나온 모델들. 초록=트리 계열, 파랑=딥러닝.

트리 계열 — 표 데이터의 주력

"질문을 나뭇가지처럼 분기(온도 20도↑? → …)"하는 결정트리를 여러 개 묶은 것들. 센서 숫자 데이터와 궁합이 좋아 스마트팜에서 많이 쓴다.

랜덤포레스트 트리
RandomForest

나무 여러 개가 투표. 작은 데이터에 안정적. 우리 데이터(600줄)에서 최다 1등 — 유명 부스팅보다 나았다.

그래디언트 부스팅 트리
GBT / Gradient Boosting

나무를 순차로 쌓아 앞 나무 실수를 다음 나무가 보정. 아래 3형제가 이 방식의 고성능 구현.

XGBoost 트리
xgboost

부스팅 대표주자. 우리 착과수 예측 1등(R² 0.92, 단 교차검증 땐 불안정). 큰 데이터일수록 강함.

LightGBM 트리
lightgbm

빠르고 가벼운 부스팅. 우리는 신뢰구간(분위수 회귀) 뽑는 데 활용.

CatBoost 트리
catboost

범주형 데이터에 강한 부스팅. 관부직경 예측 1등(0.89).

앙상블 기법
Ensemble

모델을 합침(예측 평균/결합). 트리는 인과가 약하니 LSTM과 앙상블해 보완하자는 게 회의 아이디어.

딥러닝 — 시간·서사를 다룰 때

LSTM 딥러닝
Long Short-Term Memory

기억력 있는 순차 모델. "낮에 높았다 밤에 낮음" vs 반대 같은 시간 순서·인과를 학습. 트리가 약한 부분.

트랜스포머 딥러닝
Transformer

요즘 AI(챗GPT 등)의 뼈대. 시퀀스 전체를 한눈에 보는 어텐션 방식. 히스토리 연결에 강함.

인코더-디코더 구조
Encoder-Decoder

인코더: 입력을 압축 요약. 디코더: 그걸 풀어 예측 생성. 회의의 berry2vec가 이 구조(인코더만 떼어 착과수 디코더에 붙임).

Life2vec → berry2vec 응용
life2vec

덴마크 인구 생애 데이터로 수명을 예측한 모델. "작물의 일생을 서사로" 응용한 게 berry2vec. 단점: 데이터 대량 필요(회의 우려).

대화에서 이것만 기억

"우리 데이터는 작고(수백 줄) 표 형태라 트리 계열이 주력, 시간 인과가 필요하면 LSTM을 얹는다" — 이 한 문장이면 모델 선택 논의를 따라갈 수 있다. 그리고 "유명 모델이 항상 최강은 아니다, 항목별로 돌려 비교해야"가 우리 팀이 데이터로 증명한 결론.

03

데이터 · 피처

모델보다 데이터가 점수를 더 많이 좌우한다 — 우리 팀이 몸으로 겪은 것.

용어뜻우리 예
X (피처/입력)모델이 보고 판단할 재료환경 9개(EC·pH·CO₂·온습도·일사…) + 이력
Y (타깃/정답)맞히려는 값생육 7개(착과수·초장·관부직경…)
정형 데이터표(행·열) 형태우리 CSV/엑셀 — 트리 계열이 강함
시계열시간 순서 있는 데이터1분 단위 환경 센서 로그
임베딩데이터를 숫자 벡터로 변환berry2vec의 시계열 임베딩

피처 엔지니어링 점수의 핵심

피처 엔지니어링 = X를 더 똑똑하게 가공하는 것. 새 열 추가만이 아니라 "기존 정보를 어떻게 요약하느냐"가 진짜 무기. 우리 팀 최대 성과가 여기서 나왔다.

우리가 점수 올린 실제 피처
  • 집계 기간 바꾸기: 환경을 "주 평균" 대신 "조사일 직전 N일 평균"으로 → 착과수 R² 0.69 → 0.91. 항목마다 최적 기간 다름(착과수 14일, 관부직경 7일).
  • 적산온도·누적일사·경과주: 시간·누적 정보를 피처로. (농학의 적산온도 개념을 코드로.)
  • prev(지난주값): 강력하지만 함정(4장 참고).

전처리 기본기

  • 정규화/표준화: 값 범위를 맞춤(0~1 등). 주의 — 우리 데이터는 동일값이 뭉쳐 있어 정규화로 뭉개면 정보 손실. "안 뭉개고 다 살리기"가 유리했다.
  • 결측치(NaN): 빈 값. 평균으로 채우거나(fillna) 제거.
  • 이상값(outlier): 물리적으로 불가능한 값. 우리는 2월 내부습도 13.9%(겨울 온실 불가능)를 센서 고장으로 판단해 제거 → 과습 경고가 살아남. 모델 전에 월별 스캔 필수.
  • 데이터 살리기: 측정라인5 × 표본2 = 10개를 평균으로 뭉개 60줄이던 걸, 안 뭉개 600줄로 → 성능 급등(초장 0.60→0.93). 모델 튜닝보다 큰 효과.
04

학습 · 검증 · 과적합

"점수 잘 나왔다"를 믿으면 안 되는 이유.

용어뜻
학습셋 / 검증셋 / 테스트셋
train / validation / test		공부용 / 중간점검용 / 최종시험용 데이터 분리. 시험 문제를 미리 보면 안 되듯 나눈다.
과적합 overfitting학습 데이터만 달달 외워 새 데이터엔 못 맞힘. 데이터 적은데 피처 많으면 발생(우리: 열 9→27개 늘리자 0.58→0.42 하락).
과소적합 underfitting덜 배워서 학습 데이터조차 못 맞힘.
교차검증 cross-validation데이터를 여러 조각으로 나눠 돌아가며 시험. 단일 시험의 운빨을 걸러냄.
하이퍼파라미터사람이 정하는 모델 설정(나무 개수 등). n_estimators=300 같은 것.
우리 팀이 겪은 두 함정 (대화에서 나오면 아는 척 가능)

① 데이터 누수(leakage) + GroupKFold — 같은 온실·날짜 측정이 학습셋과 시험셋에 나뉘어 들어가면 "커닝"이 된다(누수). GroupKFold는 같은 그룹을 한쪽에만 넣어 이걸 막는다. 우리 600줄 검증의 핵심 안전장치.

② prev(지난주값) 함정 — 지난주 실제값을 넣으면 점수가 확 오르지만, 대회가 "미래 여러 주"를 물으면 그 값을 못 쓴다 → 점수 폭락. 확인법: prev 빼고 돌려본다. 유지되면 진짜 실력, 뚝 떨어지면 의존(위험). 그래서 코드에 USE_PREV 플래그로 두 버전을 둔다.

단일 시험은 못 믿는다

우리 경험: "착과수 0.91, 14일 최적, XGBoost 최고"가 교차검증에선 "0.79, 7~21일 다 비슷, XGBoost 꼴찌"로 뒤집혔다. 단일 20% 시험은 운에 흔들려 뻥튀기. 반드시 교차검증 + 오차범위(±)로 보고, ± 겹치면 "차이 없음"으로 판단.

05

평가 지표

"잘한다"를 숫자로. 우리는 R²·MAE를 쓴다.

회귀 vs 분류 — 먼저 이걸 구분

  • 회귀(regression): 연속된 숫자를 맞힘. "착과수 4.8개", "초장 22cm". 우리 주 방식.
  • 분류(classification): 카테고리를 맞힘. 회의의 berry2vec는 착과수를 "3개 이하 ~ 8개 이상"의 6클래스로 분류했다.
지표쓰임방향메모
R² (결정계수)회귀↑ 좋음 (최대 1)"예측이 정답 변동을 얼마나 설명하나". 0.9=우수, 음수=엉망. 우리 착과수 0.83.
MAE (평균절대오차)회귀↓ 좋음평균 몇 개 틀리나. 착과수 MAE 1.36 = 평균 1.36개 오차(작은 값이라 큼).
RMSE회귀↓ 좋음큰 오차에 더 민감한 오차 지표.
정확도 (Accuracy)분류↑ 좋음몇 % 맞혔나. 클래스 불균형 땐 오해 소지.
F1 점수분류↑ 좋음정밀도·재현율의 균형. 불균형 데이터에 정확도보다 신뢰.
R² 높다고 중요한 게 아니다

R²는 "예측 난이도"일 뿐. 착과수 R²는 중간(0.69~0.83)이어도 대회 핵심(열매=돈)이다. 반대로 엽수는 R²가 낮아(0.06) 예측이 어렵지만 착과수와 무관해 덜 중요. 지표 순서 ≠ 중요도 순서.

06

불확실성 — 신뢰구간

"착과수 4.8개"보다 "4.8개, 3~7개 사이 80% 확신"이 훨씬 쓸모 있다. 우리 팀이 실제 구현한 고급 기법.

등각예측 / CQR — 우리가 쓰는 것

등각예측(Conformal Prediction) = 점 예측에 수학적으로 보장된 신뢰구간을 붙이는 기법. 우리는 CQR(등각 분위수 회귀) 사용 — LightGBM으로 하한·상한을 예측하고, 보정량 Q로 조정해 "명목 커버리지 80%"를 맞춘다(ALPHA=0.20). 제출도 "값 + 하한 + 상한" 형태로 나간다.

왜 중요한가 — 의사결정과 직결

  • 커버리지(coverage): 신뢰구간이 실제 정답을 담는 비율. "80% 커버리지"면 100번 중 80번은 구간 안에 정답이 든다는 뜻. 실측이 명목과 맞는지 검증한다(우리 문제3_커버리지결과.csv).
  • 구간이 좁으면 확신 있는 예측 → 과감히 결정. 넓으면 불확실 → 보수적으로 대응하거나 사람이 개입. 농작업 의사결정에 직접 쓰인다.
대화 팁

"이 예측 신뢰구간은 얼마고, 커버리지는 검증했나요?"라고 물으면 개발자가 "이 사람 좀 아는데" 한다. 단일 점 예측만 들고 오면 "구간도 같이 봅시다"가 정석 반응.

07

상관 ≠ 인과 · 예측 vs 제어

이 대회의 승부를 가르는 개념. 우리 팀 기획안의 핵심 논리이기도 하다.

한 줄 정의

상관: 두 값이 같이 움직임. 인과: 하나가 다른 하나의 원인. 같이 움직인다고 원인은 아니다. 겨울엔 습도도 높고 딸기도 익는데, 습도가 익힌 게 아니라 "겨울"이 둘 다의 배후.

왜 예측 모델로는 "제어 효과"를 못 뽑나

우리가 직접 확인한 것: 습도·급액·온도를 바꿔 착과수 변화를 시뮬(step10)했더니 거의 안 변했다. 이유:

  • 데이터가 농가의 정상 재배 관찰 기록일 뿐, "일부러 습도를 바꿔본" 실험이 아님 → 인과를 뽑을 재료가 없다.
  • prev(지난주값)가 예측을 지배해 환경 변화를 눌러버림.
  • 모델이 안 본 조합은 뭉갠다.
그래서 우리 팀의 결론 = 하이브리드

예측("이 환경이면 착과수 몇?")은 데이터로 잘 된다(R² 0.83). 하지만 제어("습도 올리면 착과수 는다?")는 순수 데이터로 못 답한다(인과 문제). → AI 예측 + 딸기 농학 규칙의 하이브리드로 간다. prev 빼고 재시뮬하면 급액EC·CO₂는 농학과 일치하는 신호를 주지만(믿을만), 습도는 계절과 얽힌 가짜 신호라 걸러야 한다 — 그 판단이 바로 농학 지식.

피처 중요도도 같은 함정

우리 모델에서 예측 중요도 1등은 외부온도(계절, 74~80%) — 그런데 못 바꾼다(하늘). 조절 가능한 것 중엔 내부습도·급액EC가 착과수를 좌우. "예측에 중요한 변수"와 "제어에 유용한 변수"는 다르다. 의사결정은 바꿀 수 있는 변수 중심으로.

08

전문가와 대화하기

알아듣기 + 되던지기. 왼쪽을 말하면 오른쪽 뜻이 전달된다.

이렇게 말하면
이런 뜻 / 효과
"이거 회귀예요 분류예요?"
숫자를 맞히는지 카테고리를 맞히는지 확인. 문제 정의를 짚는 기본기.
"교차검증 했어요? 오차범위는요?"
단일 시험 뻥튀기를 경계. "±겹치면 차이 없음"을 안다는 신호.
"prev 빼고도 점수 유지돼요?"
누수·미래예측 대비를 확인. 우리 팀 핵심 함정을 짚음.
"GroupKFold로 누수 막았어요?"
같은 온실·날짜가 학습·시험에 섞였는지. 검증 신뢰도 질문.
"신뢰구간이랑 커버리지는요?"
점 예측만 말고 불확실성까지. 의사결정용 관점.
"그거 상관이에요 인과예요?"
제어 효과라고 주장할 때 결정타. 관찰 데이터의 한계 지적.
"조절 가능한 변수 중에선 뭐가 중요해요?"
외부온도(못 바꿈) 말고 실제 의사결정에 쓸 변수로 방향 전환.
"이 항목은 어떤 모델이 제일 나았어요?"
"유명 모델=최강 아님, 항목별 비교" 원칙을 공유.
듣다가 막히면 되물을 3종 세트

"그게 X예요 Y예요?"(개념 구분) · "그럼 그게 왜 중요해요?"(의미 확인) · "그거 검증은 어떻게 했어요?"(신뢰도 확인). 이 3개면 대부분의 ML 대화를 따라가고 이끌 수 있다.

모르면 이렇게 (아는 척 금물)

ML은 반례가 많은 분야다. 모르면 "그 용어 우리 프로젝트 맥락에서 한 문장으로 설명해줄래요?"가 제일 안전하고 빠르다. 우리 팀 노하우.md·submit_pipeline.py를 같이 보며 "우리 경우엔?"으로 좁히면 대화가 구체화된다.

용어집

빠른 참조. 회의·코드에 나온 순.

피처 feature (X)
모델 입력 변수. 우리는 환경9 + 이력 + prev.
타깃 target (Y)
맞히려는 정답. 우리는 생육 7개.
피처 엔지니어링
X를 똑똑하게 가공. "직전 N일 평균"이 최대 무기였다.
회귀 / 분류
숫자 맞히기 / 카테고리 맞히기.
랜덤포레스트
나무 투표 트리 모델. 작은 데이터에 안정적.
부스팅 XGBoost·LightGBM·CatBoost
나무를 순차로 쌓아 보정. 큰 데이터에 강함.
앙상블
여러 모델을 합쳐 성능·안정성↑.
LSTM
기억력 있는 딥러닝. 시간 순서·인과 학습.
트랜스포머
어텐션 기반 딥러닝. 최신 AI의 뼈대.
인코더-디코더
입력 압축(인코더) → 예측 생성(디코더).
임베딩
데이터를 숫자 벡터로 변환.
정규화 / 표준화
값 범위 맞추기. 우리 데이터는 과도하면 정보 손실.
결측치 NaN
빈 값. 채우거나 제거.
이상값 outlier
불가능한 값(2월 습도 13.9%). 제거 대상.
과적합 overfitting
학습 데이터만 외워 새 데이터에 실패.
교차검증 cross-validation
조각내 돌아가며 시험. 운빨 제거.
GroupKFold
같은 그룹을 한쪽에만 넣어 데이터 누수 방지.
데이터 누수 leakage
시험 정보가 학습에 새어드는 것. 점수 뻥튀기 원인.
하이퍼파라미터
사람이 정하는 모델 설정(나무 수 등).
R²
회귀 정확도(↑좋음, 최대 1). 변동 설명력.
MAE / RMSE
평균 오차 지표(↓좋음).
F1
분류 균형 지표. 불균형 데이터에 신뢰.
등각예측 / CQR conformal
예측에 보장된 신뢰구간을 붙임. 우리가 구현.
커버리지
신뢰구간이 정답을 담는 비율. 80% 명목.
상관 ≠ 인과
같이 움직임 ≠ 원인. 제어 논의의 핵심.
피처 중요도
어떤 X가 예측에 크게 기여했나. "중요"≠"조절 가능".
하이브리드
AI 예측 + 농학 규칙 결합. 우리 대회 전략.

회의록 1·2·3 + 팀 코드(README·노하우.md·submit_pipeline.py) 기반 재구성. 파란 박스는 우리 프로젝트가 실제 사용/검증한 내용. 수치(R²·MAE·커버리지 등)는 팀 기록값으로, 데이터·검증 조건에 따라 달라질 수 있다. 개념 설명은 대화 이해용으로 단순화했으며 엄밀한 정의는 개발자와 확인 권장.
